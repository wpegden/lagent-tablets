# New TODO

1. Done: Clean up supported `.tex` statement environments for tablet nodes.
The current deterministic checker accepts `{theorem}`, `{lemma}`, `{definition}`, and `{proposition}` for ordinary tablet nodes, and only `{definition}` / `{proposition}` in `Preamble.tex`. We should change that policy so the main tablet accepts `{corollary}` as a first-class statement environment and forbids `{proposition}` outside the preamble. `Preamble.tex` may continue to use `{definition}` and `{proposition}` for imported Mathlib-facing interface material. This should be handled as a checker/policy cleanup rather than something we try to rewrite mid-run in the current `extremal` project.

2. Done: Make `Preamble.tex` correspondence first-class rather than only a global invalidation input.
At the moment, `Preamble.tex` participates in correspondence only indirectly: its contents are mixed into node correspondence fingerprints, so changes to the preamble reopen theorem-stating correspondence, but the preamble entries themselves are not surfaced as first-class correspondence objects with their own verdicts. We should model preamble definitions/propositions as explicit correspondence-checkable interface items, so imported Mathlib-facing definitions in `Preamble.tex` are actually reviewed on their own terms rather than only causing a global frontier reopen.

3. Done: Reconsider the initial theorem-stating node-count guidance and likely widen it to 15-50 nodes.
The current broad theorem-stating prompt says “Aim for 10-30 nodes depending on the paper's complexity.” For papers with a substantial but still manageable result DAG, that can push workers toward an overly compressed first slice. We should consider updating the guidance to something like 15-50 nodes, while still emphasizing that the decomposition should be paper-faithful and not gratuitously over-fragmented. This is a prompt-policy question rather than a change we need to make mid-run.

4. Done: Theorem-stating retry prompts should surface the previous invalid blocker explicitly.
When a theorem-stating worker attempt fails deterministically and the supervisor retries the same in-flight cycle, the next worker prompt currently restarts from the ordinary theorem-stating prompt and does not mention why the previous attempt failed. That is a real context gap: the worker may retain session memory, but the prompt does not say that the prior attempt was invalid, what specific blocker caused rejection, or whether the worktree was rolled back. Retry prompts for theorem-stating should include a clear “previous attempt failed because …” section, at least for repeated attempts on the same in-flight cycle.

5. Done: Fix the orphaned `REJECTED` outcome semantics in proof-formalization.
The architecture distinguishes `INVALID` (deterministic failure), `NO_PROGRESS` (clean but unproductive), and `REJECTED` (content rejected by verification/review). In the live proof-formalization path, `INVALID` and `NO_PROGRESS` are real outcomes, but `REJECTED` is mostly an orphaned category: it is documented and referenced in prompt/reviewer logic, yet the active proof-worker path rarely or never emits it as a distinct cycle outcome. We should either implement a real `REJECTED` path for verification-level rejection in proof-formalization or simplify the architecture/docs so the outcome model matches the actual control flow.

6. Done: After repeated consecutive `INVALID` attempts, explicitly prod the reviewer to consider resetting to a valid checkpoint.
The right reset model is reviewer-controlled, not automatic. But if the same cycle keeps hitting consecutive deterministic `INVALID` outcomes, the reviewer should be reminded that the current path may be unproductive. A reasonable first policy is: every 5th consecutive `INVALID`, add an explicit reviewer prompt note to consider resetting to one of the available valid checkpoints. The supervisor should enforce that reset targets may only be chosen from valid checkpoints; arbitrary dirty states or invalid attempts must not be legal reset destinations. This should remain a prompt/escalation nudge, not an automatic rollback.

7. Done: Add flowchart documentation under `flows/` to make cycle control flow explicit.
Once the current outcome/reset cleanup is implemented, we should add concise markdown documents under `flows/` with simple ASCII flowcharts for each major control-flow path: theorem-stating, proof-formalization, cleanup, verification gating, retry/invalid handling, checkpoint/reset behavior, and phase transitions. Each file should show inputs, decision points, possible outcomes, and a brief note on when that path applies. The goal is to keep the live control-flow model legible and synchronized with the code, not to write long prose.
