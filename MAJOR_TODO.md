# Major TODO

1. [x] Preserve Codex session continuity for stateful roles.
Stateful Codex roles like theorem-stating/proof workers and reviewers currently run through `codex exec --ephemeral` in `lagent_tablets/agents/codex_headless.py`, so they lose conversation context across bursts even when the system semantics expect continuity. We should persist the Codex session/thread id from one burst, stop using ephemeral mode for stateful roles, and resume later bursts with `codex exec resume <session-id> ...`. Verification agents can remain fresh.

2. [x] Remove or redesign hard wall-clock burst timeouts in headless backends.
The documented policy says we want to avoid aggressive timeouts that kill long-running useful work, but the headless backends still impose hard completion deadlines. `codex_headless.run(...)` enforces a supervisor-side `deadline_exit`, and `script_headless` is stricter still: it embeds an in-script watchdog based on `AGENT_TIMEOUT_SECONDS` and also has a supervisor-side deadline loop. We need a clear policy for which roles, if any, should have hard wall-clock limits, and should likely move long-running roles to liveness/inactivity-based termination instead.

3. [x] Soundness continuity is documented more strongly than it is implemented.
The docs say verifiers receive their own previous findings so they can detect whether claimed fixes are genuine. That is true for correspondence, where prior agent-specific results are loaded and injected into `build_correspondence_prompt(...)`. It is not true in the same way for per-node soundness: `build_node_soundness_prompt(...)` supports `previous_issues`, but `_run_single_node_soundness(...)` never passes them, and there is no analogous load-and-inject path for prior soundness findings. Accepted soundness results are reusable via fingerprint matching, but that is cache reuse rather than prompt continuity.

4. [x] Theorem-stating is under-checked on the normal path.
Outside reviewer-authorized `restructure` mode, theorem-stating mainly relies on worker handoff schema validation, repair/scope checks where applicable, a Preamble-specific definition check, and `lake build Tablet`. That means deterministic checks already implemented in `check.py` are not uniformly enforced on the common theorem-stating path: declaration-name versus filename agreement, marker versus filename agreement, `.tex` format validation, import allowlist enforcement across touched nodes, and axiom audit. The phase where structural artifacts are born should apply an appropriate deterministic tablet check even on the ordinary path, not only in restructure mode.

5. [x] Proof-formalization should keep the validator refactor but recover the stronger hard-mode checks.
The active proof-formalization validator (`validate_worker_cycle_v2(...)`) is weaker than the older snapshot-based validator that still remains in `cycle.py`. In particular, for newly created helper nodes it does not enforce declaration-name versus filename agreement, `.tex` format validation, compilation, axiom audit, or broader tablet consistency. The right fix is not to revert to duplicated legacy validation logic, but to preserve the intended refactor: workers and the supervisor should both use the same deterministic checking scripts/functions, and those shared checks should be strong enough for hard mode and new-node creation.

6. [x] Newly closed proof-formalization nodes can bypass correspondence and paper-faithfulness.
After a proof worker burst, the proof-formalization loop currently verifies newly created nodes only if they remain open, plus the active node only if it changed and remains open. That means a brand-new helper node can be created and fully proved in Lean in the same cycle, become closed immediately, and enter the tablet without ever going through correspondence or paper-faithfulness verification. Skipping NL-proof soundness in that situation is defensible; skipping correspondence/paper-faithfulness is not.

7. [x] Node kind assignment is too heuristic for how much downstream logic depends on it.
The system currently infers new node kind using crude heuristics: theorem-stating guesses `paper_main_result` versus `paper_intermediate` from substrings in the filename, and proof-formalization registers new nodes as `helper_lemma`. But `kind` affects orphan detection, reviewer prompt framing, viewer semantics, and overall structural interpretation of the tablet. These classifications should be intentional parts of the early theorem-stating package, not naming accidents discovered later.

8. [x] “Single source of truth” for deterministic checking is not fully true yet.
`lagent_tablets/check.py` is supposed to be the canonical implementation of deterministic checking, but `lagent_tablets/verification.py` still contains separate `check_node(...)` and `check_tablet(...)` logic as well as script-generation machinery, and core runtime paths still import helper pieces like `write_scripts`, `_is_lake_package_error`, and `FORBIDDEN_KEYWORDS_DEFAULT` through that legacy module. This is not harmless duplication: the old checker logic differs materially from `check.py`, creating classic drift risk in the exact part of the system that most needs one canonical implementation.

9. [x] Allow paper-facing imported definitions in `Preamble.tex`.
The current model forces `Preamble.lean` to contain only imports and also restricts `Preamble.tex` to proposition environments, which blocks a common and important use case: paper-facing definitions that correspond directly to imported Mathlib definitions or notation. In typical projects, many basic definitions belong in this imported preamble layer rather than as separate tablet nodes. We should allow `definition` environments in `Preamble.tex`, model them as correspondence-checkable imported definitions, and clarify how correspondence is evaluated for this preamble-only definition layer.

10. [x] Human-reviewed main-result correspondence should be sticky until renewed human review.
After a human has reviewed and accepted the paper-facing package, that establishes a specially trusted correspondence relationship for the reviewed main results. If correspondence is later lost for any `paper_main_result` node, whether by direct statement changes or via the recursive correspondence invalidation rules, the system should require renewed human review before treating the package as trusted again. This should be modeled explicitly rather than relying on ordinary automated correspondence rechecks alone.

11. [x] The system needs an explicit "paper has a fundamental gap" escalation path during early theorem-stating.
Right now theorem-stating assumes the paper-facing NL proof/DAG process will eventually succeed with enough repair or restructuring. That leaves no honest response mode for the case where, during the initial coarse theorem-stating stage, a worker concludes that the source paper itself appears to contain a genuine and likely unfixable mathematical gap. We should add a distinct worker status such as `CRISIS`, limited to early theorem-stating, meaning roughly "the paper seems to have a fundamental gap that cannot be repaired within the intended formalization." That signal should be reviewed by an appropriate reviewer/verification path, and if accepted it should generate an explicit human-input event rather than being treated as an ordinary local failure or restructure request.

12. [x] Rewind should be commit-only, clean, and supported by finer committed checkpoints.
The current rewind/replay model has been too error-prone because it allows partial historical surgery: some runtime artifacts get rewound while others retain newer state, and the resulting mixed provenance is then interpreted by current code. We should simplify rewind so that it only targets explicit git commit/tag checkpoints and restores them exactly with a clean git reset/clean operation, rather than trying to preserve or reconstruct partial state. If we want finer-grained resumption, the right answer is to create more committed checkpoints, for example after each subcycle boundary such as worker completion, verification completion, and reviewer completion, so that replay remains exact without ad hoc state mutation.

13. [x] The web viewer should read canonical project artifacts directly instead of mirrored static copies.
The current viewer architecture still depends on copying live repo artifacts into a separate static web directory, which has repeatedly created stale-state, permission, and rewind-sync bugs. We should remove the artifact-mirroring layer and have the website read the formalization project directly through a thin backend that serves canonical current-state files and historical snapshots from git. Current view should come from the live repo artifacts; history should come directly from committed cycle checkpoints/tags. That would eliminate the extra sync process and greatly reduce the number of viewer-specific failure modes.

14. [x] Persistent agentapi session reuse is keyed too weakly.
`agentapi_backend.ensure_server(...)` currently reuses persistent worker/reviewer sessions based on provider only. That is too weak for a system that allows model overrides, phase-specific role changes, and fallback adjustments: a reused Gemini or Claude session can silently ignore an intended model/configuration change unless it is explicitly restarted. Session identity should include the effective runtime configuration, at least provider plus model and relevant options, with deliberate handling for safe in-session switches such as Gemini `/model` where supported.

15. [x] agentapi send-failure log fallback points at the wrong file.
On prompt-send failure, `agentapi_backend.run(...)` tries to inspect `.agent-supervisor/logs/agentapi-reviewer.log`, but the actual server logs are written as `agentapi-{role}-{port}.log`. That means the fallback error path is likely looking in the wrong place and can miss the real startup/send failure diagnostics. It is a smaller issue than the session-identity bug, but still worth fixing because this path matters when diagnosing exactly the kinds of flaky backend failures we have been dealing with.

16. [x] `proof_complete_style_cleanup` should become a real terminal cleanup phase, not a relabeled proof cycle.
Right now `proof_complete_style_cleanup` is not operationally distinct enough from `proof_formalization`; both still run through the same proof-cycle machinery. That is the wrong semantics. Cleanup should begin only after the tablet is already mathematically complete and accepted. Its purpose is optional semantics-preserving polish only: formatting, harmless refactors, import tidying, comments, and similar cleanup. If a cleanup attempt tries to make a semantic change, the system should reject that cleanup attempt rather than reopening earlier phases. After bounded cleanup attempts, the supervisor should be able to stop successfully at the last good proof-complete state even if cleanup never lands. In other words, cleanup should be a true terminal polish phase over an already accepted final DAG, with validation specifically designed to enforce semantic preservation.

17. [x] `check.py` must be truly canonical and invoked consistently on every artifact-creating path.
This needs to be an explicit invariant, not an aspirational comment. Every deterministic validation that can block or reject work should come from `lagent_tablets/check.py`, and every path that creates or mutates proof-tablet artifacts should invoke those canonical checks consistently. There should be no hidden deterministic gates implemented ad hoc in supervisors, validators, or helper modules that agents cannot see or run themselves. If a deterministic rule matters, it should live in `check.py`, be surfaced to agents, and be applied uniformly across theorem-stating, proof-formalization, cleanup, reviewer acceptance, and any other artifact-producing path.

18. [x] The viewer should support multiple formalization projects via path-based routing.
Right now the viewer effectively assumes a single project, which does not scale once we have multiple formalization repos on the same server. A simple and robust model is path-based routing, for example `/lagent-tablets/extremal/` for the `extremal` project, with the backend resolving that path segment to the corresponding formalization repo and serving current state/history from that project's canonical artifacts. This should work for both the live view and cycle history, and it fits naturally with the broader plan to remove the extra mirrored-static-state layer.

19. [x] Theorem-stating should support a scoped Lean shortcut when a node can be closed immediately.
Sometimes a theorem-stating worker can see that the currently authorized target can be proved in Lean immediately from existing children, making another NL-proof repair cycle wasteful. We should support an explicit theorem-stating Lean shortcut: within the currently authorized theorem-stating scope only, the worker may close the target directly in Lean instead of merely improving the NL proof, provided the prompt includes the exact deterministic check command the supervisor will use and the supervisor treats a successful Lean closure as superseding the need for further NL-proof repair on that node. This should not silently collapse theorem-stating into general proof-formalization; it should remain a scoped exception tied to the currently authorized target/work slice.

20. [x] Theorem-vs-definition authoring should be an explicit structural policy, not just informal guidance.
Because correspondence propagation and paper structure now treat definitions differently from lemmas/theorems/corollaries, it matters whether a concept is introduced as a genuine definition node or smuggled into a result statement. The system should make this an explicit theorem-stating policy: workers should introduce paper-facing concepts as definition nodes when that is what they are, and reviewers should be willing to reject or reshape theorem-stating packages that use theorem/lemma/corollary nodes as disguised definitions. This is more than prompt style; it is a structural modeling norm that affects correspondence behavior and DAG quality.
